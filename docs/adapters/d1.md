# Cloudflare D1 Adapter

D1 is Cloudflare's serverless SQLite database. It runs at the edge via Cloudflare Workers/Pages.

> **Important**: D1 requires deploying to **Cloudflare Pages**, not Vercel. If you use Vercel, choose SQLite, Turso, or Postgres instead.

## Prerequisites

- A Cloudflare account
- `wrangler` CLI: `npm install -g wrangler`
- `wrangler login`

## Create a D1 database

```bash
wrangler d1 create my-app-db
```

Add the output binding to `wrangler.toml`:

```toml
[[d1_databases]]
binding = "DB"
database_name = "my-app-db"
database_id = "your-database-id"
```

## .env configuration

```env
DB_ADAPTER=d1
D1_DB_NAME=DB
```

> D1 does not use `DATABASE_URL` — the binding is injected by the CF runtime.

## Wiring the binding in Next.js on Cloudflare Pages

In your Cloudflare Pages Functions entry point (e.g. `functions/_middleware.ts`), set the global binding before the Next.js handler runs:

```typescript
export async function onRequest(context) {
  (globalThis as any).__D1_BINDING__ = context.env.DB
  return context.next()
}
```

## First-time setup (local dev)

```bash
wrangler d1 execute DB --local --file=<(pnpm drizzle-kit generate --config=drizzle.config.ts)
```

Or use `pnpm db:setup` with wrangler's local D1 emulation configured.

## Running migrations (production)

```bash
pnpm db:generate         # generate SQL in ./drizzle/d1/
wrangler d1 migrations apply DB --remote
```

## Production considerations

- D1 is free up to 5GB and 25 million reads/day
- Latency is lowest for read-heavy workloads close to CF edge nodes
- Write performance is limited by D1's single-region primary write constraint
